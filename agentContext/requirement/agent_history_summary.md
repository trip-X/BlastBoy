# Agent 历史记录提炼（Blast Boy）

本文档用于汇总当前会话中已经确定的关键需求、约束、流程与可复用提示词产物，便于后续交给不同端（开发端/美术端）继续推进。

---

## 1. 项目基本信息（已定稿）

- 游戏名称：Blast Boy
- 引擎：Cocos Creator 3.8.x
- 语言：TypeScript
- 类型：俯视角生存清版射击（简化版 Brotato）
- 项目状态：已存在 Cocos 项目，直接在项目目录内开发（无需新建项目）

---

## 2. 核心玩法范围（极简新手友好）

### 2.1 必做功能

- 玩家 WASD 键移动
- 玩家自动锁定并攻击最近敌人
- 敌人按波次生成，追击玩家，接触造成伤害
- 玩家生命值为 0 → 游戏结束
- UI 显示：血量、当前波数、游戏时长

### 2.2 明确不做（当前阶段）

- 无商店、无天赋、无多武器、无随机道具
- 不做复杂系统，保证逻辑直白、易读、易调试

> 重要补充：虽然当前不做复杂系统，但代码必须具备良好扩展性，后续可能会加。

---

## 3. 关键技术定稿（非常重要）

- 分辨率：`1280×720`（16:9 横屏）
- 平台：仅 `Web + 键盘（WASD）`
- 美术：像素风（Pixel Art）
- 像素网格基准：`32px` 网格
- 相机：跟随玩家
- 碰撞/伤害：脚本“圆形距离判定”（禁止 Collider2D / 物理碰撞工作流）

---

## 4. 代码与架构硬性要求（学习向 + 可扩展）

- 轻量但标准的模块化架构，适合新手学习
- 命名语义化、结构清晰，避免晦涩嵌套与炫技
- 关键逻辑逐行中文注释（输入、移动、索敌、生成、伤害、回收等）
- 模块合理拆分：玩家、敌人、子弹、游戏主控、对象池、UI 管理
- 低耦合、高可扩展：未来新增内容尽量“新增文件/配置”为主，避免重构核心

---

## 5. 开发执行流程（阶段门禁：必须人工验收）

### 5.1 阶段式交付原则

- 分阶段循序渐进开发，一步一个可运行小功能，不跳步
- 每个阶段结束必须提供“Cocos 界面操作步骤 + 验收清单”
- 你需要在 Cocos Creator 里亲自验证；验证后按固定格式反馈：
  - `Stage N 通过`
  - `Stage N 不通过：{现象描述}+{Console报错全文}+{相关节点/脚本线索}`
- 在你确认前，开发端不得进入下一阶段

### 5.2 开发端输出强制结构

- A. 本阶段完成内容（简短）
- B. 完整可运行的 TS 代码（列出新增/修改文件路径）
- C. Cocos Editor 操作步骤（可照做：菜单路径、点击顺序）
- D. 节点结构与拖拽绑定表（字段→拖哪个节点/资源）
- E. 验收清单 + 反馈格式
- F. 美术需求清单（仅规格，不写 AI 提示词）

---

## 6. 开发端与美术端职责分离（非常重要）

### 6.1 开发端（只做代码与编辑器搭建）

- 只负责：TS 代码、Prefab/节点结构、属性拖拽绑定说明、可运行版本
- 不负责：生成美术提示词、不生成图片、不假设资源已存在
- 当美术资源未到位：必须提供占位方案（纯色 Sprite/Label 等）并说明替换入口

### 6.2 美术端（只产出即梦AI提示词与素材规范）

- 输入：开发端输出的“美术需求清单”（用途/尺寸/命名/数量/透明/序列帧/平铺等）
- 输出：可直接复制到即梦AI的提示词 + 负面提示词 + 生成建议参数 + 验收标准
- 不做：代码、不做 Cocos 节点搭建

## 8. 像素风素材统一规范（摘录要点）

- 视角：俯视角（top-down）
- 网格：32px 对齐（尺寸尽量为 32 的倍数）
- 角色/敌人：推荐 `128×128`（或同档位统一）
- 子弹：推荐 `32×32`
- 命中特效：推荐 `64×64`，4~6 帧
- 地面平铺：推荐 `256×256`，可无缝平铺
- 禁止：文字、水印、logo、复杂背景杂物
