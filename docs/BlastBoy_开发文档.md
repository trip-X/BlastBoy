## 1. 项目目标与边界

### 1.1 核心目标（必须实现）

- 玩家用 `WASD` 移动（俯视角）
- 玩家自动锁定并攻击距离最近的敌人
- 敌人按波次生成，追击玩家，碰撞造成伤害
- 玩家有生命值，生命值为 0 则游戏结束
- UI 显示：血量、当前波数、游戏时长

### 1.2 明确不做（避免复杂化）

- 不做商店/天赋/多武器/随机道具
- 不做复杂数值系统与大量敌人种类
- 不做联网/存档/排行榜

### 1.3 体验关键词

- 新手友好：逻辑直白、容易断点调试
- 低耦合：模块边界清晰，后续加内容不推倒重来
- 可运行成品：任何阶段都能点击运行并体验

## 3. 项目目录规范（学习向、标准化、可扩展）

> 说明：当前仓库里未看到 `assets/` 目录（Cocos 通常会有）。后续写代码与搭场景时，我们会在项目内创建并按本结构组织。

建议在 `assets/` 下建立如下结构（只保留必要内容）：

- `assets/`
  - `scenes/`：场景（`Game.scene`、`Boot.scene`）
  - `prefabs/`：预制体（玩家/敌人/子弹/UI）

    - `ui/`：UI 层（UI 管理、面板脚本）
    - `core/`：通用基础设施（事件、对象池、工具、配置）
  - `art/`：美术资源（图片、图集、材质）
  - `audio/`：音频资源
  - `fonts/`：字体资源

## 4. 模块拆分与职责边界（硬性要求）

目标：每个模块“只做一件事”，互相通过明确接口协作。

### 4.1 Gameplay（玩法域）

- `Player`：
  - 读取输入并移动
  - 维护生命值（受伤/死亡）
  - 自动瞄准最近敌人（只输出目标，不直接管理敌人列表）
  - 自动开火（只负责生成子弹/发射事件）

- `Enemy`：
  - 接收玩家位置并追击
  - 碰撞/接触后对玩家造成伤害
  - 死亡时回收（对象池）

- `Bullet`：
  - 朝目标方向直线飞行
  - 命中敌人造成伤害
  - 超出距离/时间后回收（对象池）

- `WaveSpawner`（波次生成器）：
  - 只负责“何时生成、生成多少、从哪生成”
  - 不负责敌人 AI
  - 输出：当前波数、波次进度

### 4.2 Core（基础设施）

- `ObjectPool`（对象池）：
  - 统一管理预制体实例化与回收
  - 支持：预热数量、最大数量、回收重置

- `EventBus`（事件总线，可选）：
  - 用于 UI 与玩法解耦（例如“血量变化”“波数变化”“游戏结束”）
  - 要求：事件名统一、参数结构固定

- `GameConfig`（配置集中管理）：
  - 放玩家速度、子弹速度、敌人速度、波次数量等
  - 目标：所有数值可在一个文件调

### 4.3 UI（表现层）

- `UIManager`：
  - 只负责打开/关闭 UI 面板、更新显示
  - 不参与玩法逻辑

- `HUD`（抬头显示）：
  - 显示血量/波数/计时
  - 通过事件或数据绑定更新

### 4.4 可扩展性边界（强约束）

为了保证“现在极简、未来好加功能”，模块之间必须遵守下面的边界（这会直接影响你后续加商店/天赋/多武器/道具时是否需要重构）：

- **UI 不直接依赖玩法内部对象**：UI 只订阅事件或读取只读状态，不去拿 `Enemy` 列表、不去调用 `Player` 的内部方法
- **玩法不依赖 UI**：玩法只负责产生数据与事件，不关心 UI 怎么显示
- **服务可替换**：输入/音频/存档/计时等做成“服务接口 + 默认实现”，未来替换（例如键盘→手柄、Web→移动）不动核心玩法
- **配置数据驱动**：波次、敌人、武器（未来）都以配置为唯一来源，脚本只解释配置，不在代码里写死大量 if/else
- **扩展只新增文件为主**：新增内容尽量通过“新增配置 + 新增实现类”完成，避免到处改旧文件

## 5. 核心数据流（新手容易调试的“单向流”）

建议采用“玩法层产生数据 → 事件/接口通知 → UI 层显示”的单向流：

1. `Player` 生命值变化 → 触发 `HP_CHANGED` → `HUD` 更新
2. `WaveSpawner` 进入新波次 → 触发 `WAVE_CHANGED` → `HUD` 更新
3. `GameController` 计时更新（每秒一次）→ 触发 `TIME_CHANGED` → `HUD` 更新
4. `Player` 死亡 → 触发 `GAME_OVER` → `UIManager` 打开 GameOver 面板

这样做的好处：

- 断点更好打：数据从哪来一眼能追
- 后续扩展简单：加新的 UI 显示，不需要改玩法内部

### 5.1 为扩展预留的“事件/状态”清单（不实现，仅规定口径）

现在我们只会用到 HP/波次/计时/死亡，但为了未来扩展（商店、天赋、多武器、道具），建议提前把事件口径定好，后续新增时不会乱：

- `HP_CHANGED`：`{ current: number; max: number; delta: number }`
- `WAVE_CHANGED`：`{ waveIndex: number }`
- `TIME_CHANGED`：`{ seconds: number }`
- `GAME_OVER`：`{ reason: 'hp_zero' | 'debug' }`
- `ENEMY_KILLED`（未来）：`{ enemyId: string }`
- `GOLD_CHANGED`（未来商店）：`{ current: number; delta: number }`
- `WEAPON_CHANGED`（未来多武器）：`{ slot: number; weaponId: string }`

## 6. 编码规范（TypeScript + Cocos Creator）

### 6.1 命名

- 类名/组件名：`PascalCase`，例如 `PlayerController`
- 变量/函数：`camelCase`，例如 `moveSpeed`
- 常量：`UPPER_SNAKE_CASE`，例如 `MAX_ENEMY_COUNT`
- 资源命名：见“素材规范”章节

### 6.2 文件与模块

- 一个文件只放一个主要类（辅助类型可放同文件）
- 避免“万能管理器”上帝类：管理器只做聚合与调度，不塞细节
- 严禁深层嵌套：优先早返回（early return）

### 6.3 面向扩展的代码约束（学习向重点）

下面这些约束会让“后续新增系统”更容易做到“不改或少改旧代码”：

- **依赖方向固定**：`core/services/configs/types` → `gameplay` → `ui`（只能单向依赖，禁止反向引用）
- **对外只暴露最小接口**：例如 `Player` 对外只暴露 `takeDamage()`、`getPosition()`，不要让别的模块随意改内部字段
- **使用组合而非继承**：未来武器/天赋建议用“组件化 + 配置驱动”，避免层层继承导致难调试
- **避免硬编码字符串**：事件名、配置 id 统一集中定义（放 `types/`）
- **管理器可测试**：尽量把“纯逻辑”写成不依赖节点的类（接收数据、返回结果），组件只负责把逻辑接到引擎生命周期

### 6.4 注释规范（学习向硬性要求）

- **关键逻辑逐行中文注释**（输入、移动、索敌、生成、碰撞、伤害、回收）
- 每个组件类顶部写清“职责、依赖、对外输出”
- 注释写“为什么这么做”，不要只重复代码字面意思

### 6.5 日志与调试

- 关键状态变化要有 `console.log`（例如：进入新波次、玩家死亡）
- 日志必须可开关（后续用一个 `DebugConfig.enabled` 控制）
- 禁止每帧输出日志（会卡）

## 7. 场景与节点规范（可复制的搭建方式）

### 7.1 场景划分

- `Boot.scene`：资源预加载（可选，简单项目也可省略）
- `Game.scene`：主玩法

### 7.2 Game.scene 节点树建议

> 这是“建议结构”，分辨率已固化为 `1280×720`，相机跟随玩家。

- `Canvas`（UI 根节点）
  - `HUD`（血量/波次/计时）
  - `GameOverPanel`（默认隐藏）
- `World`（世界根节点）
  - `Player`
  - `Enemies`（敌人容器）
  - `Bullets`（子弹容器）
  - `VFX`（特效容器，可选）
- `MainCamera`（相机节点）
- `GameController`（挂主控脚本的空节点）

### 7.3 预制体（Prefab）规范（为后续扩展做准备）

- 每个可复用实体（玩家/敌人/子弹/UI 面板）都做成 Prefab
- Prefab 根节点只挂“入口组件”（例如 `EnemyView`），不要把逻辑散落在子节点
- 子节点只做表现：`Sprite`、`Animation`、`Label` 等
- 未来新增敌人类型时：优先复用同一个 Prefab，通过配置切换贴图/速度/血量，而不是复制粘贴一堆 Prefab

### 7.4 组件依赖注入（新手友好）

- **一律用编辑器拖拽绑定**（`@property(Node)`）
  - 好处：运行前就能发现引用为空的问题
- 避免运行时全局查找：禁止大量使用 `find()`

## 8. 分阶段开发流程（一步一个可运行版本）

每个阶段都要满足：

- 能运行
- 有最小可玩性
- 有对应验收清单（下面给出）

### 阶段 0：项目检查与最小场景

- 目标：能打开项目、创建 `Game.scene`、点击运行显示一个背景与玩家占位
- 验收：运行无报错；玩家节点可见

### 阶段 1：玩家 WASD 移动

- 目标：玩家用键盘移动；相机能跟随玩家
- 验收：按键响应流畅；松开立即停止；无抖动

### 阶段 2：敌人追击（单个）

- 目标：生成 1 个敌人，持续朝玩家移动
- 验收：敌人能稳定追击；碰撞判定伤害稳定；无明显抖动

### 阶段 3：自动索敌 + 子弹攻击

- 目标：玩家自动寻找最近敌人，定间隔发射子弹
- 验收：最近目标变化正确；子弹方向正确；命中扣血/死亡

### 阶段 4：波次生成与基础 UI

- 目标：按波次生成多个敌人；HUD 显示 HP/波数/计时
- 验收：波数递增正确；计时正确；HP 变化正确

### 阶段 5：游戏结束与重开

- 目标：HP=0 出现 GameOver；可重开
- 验收：死亡后不再生成/攻击；重开后状态清零

## 9. 素材规范（极梦AI生产：尺寸、命名、风格一致性）

### 9.1 通用规范

- 图片格式：`PNG`（透明背景）
- 颜色空间：sRGB
- 单张尺寸：建议 `64/128/256/512` 等 2 次幂（便于图集）
- 像素网格：本项目固化为 `32px` 网格（角色/敌人/地面/UI 尽量对齐 32 的倍数）
- 锚点：角色/敌人/子弹统一使用“中心点”（后续做动画与旋转更省心）
- 命名：`类别_名称_状态_尺寸`，例如：`char_blastboy_idle_256.png`

### 9.2 美术风格（已确认：像素风）

你已确认本项目使用：**像素风（Pixel Art）**。

#### 像素风“风格包基底提示词”（可复制）

```text
像素风（pixel art），俯视角（top-down），清晰像素网格，强轮廓，高对比，简单阴影，纯色块上色，复古街机风，画面干净，无文字无水印，透明背景PNG（如果是角色/子弹/UI）
```

#### 像素风输出尺寸建议（32px 网格）

- 角色/敌人：`96×96`（约 3×3 格）或 `128×128`（约 4×4 格）
- 子弹：`32×32`（1×1 格）或 `64×64`（2×2 格，带尾迹更清晰）
- 命中特效：`64×64`（2×2 格，做 4~6 帧）
- 地面平铺：`256×256`（8×8 格）

> 这是“建议值”，不影响架构；后续你想把角色做得更大/更小也很好改。

### 9.3 像素资源“可扩展”规范（避免未来返工）

像素风最容易踩坑的是“后期资源不统一，UI 变糊、角色比例乱”。为避免返工，约束如下：

- **所有 UI 尺寸优先用 32 的倍数**（你已确认 32px 网格）
- **角色/敌人尺寸档位固定**：推荐只选 `96×96` 或 `128×128` 作为全项目档位
- **同类资源同一轮廓粗细**：角色描边与敌人描边尽量一致
- **调色板尽量统一**：每类资源不超过 8~16 种主色（越少越统一）
- **透明边距统一**：同一角色不同状态保持相同画布尺寸与边距（未来加走路/受击动画不会跳）

## 16. 技术实施规格（待补充，作为最终落地口径）

> 说明：本章为技术审查要求新增的“落地口径”章节。需求方/技术负责人补齐后，开发需严格按此实现与验收。

### 16.1 内部接口（事件或状态）

- 采用方案：A EventBus
- 事件清单与契约：
  - `HP_CHANGED`：`{ current: number; max: number; delta: number }`
    - 发布者：`Player`
    - 订阅者：`HUD`
    - 触发时机：每次 HP 发生变化时触发（受伤/治疗/重置）
  - `WAVE_CHANGED`：`{ waveIndex: number }`
    - 发布者：`WaveSpawner`
    - 订阅者：`HUD`
    - 触发时机：进入新波次时触发（波次索引从 1 开始）
  - `TIME_CHANGED`：`{ seconds: number }`
    - 发布者：`GameController`
    - 订阅者：`HUD`
    - 触发频率：每秒一次
    - 暂停口径：失焦暂停时不递增
  - `GAME_OVER`：`{ reason: 'hp_zero' | 'debug' }`
    - 发布者：`Player`（`hp_zero`）或 `GameController`（`debug`）
    - 订阅者：`UIManager`
    - 触发时机：玩家死亡后仅触发一次

### 16.2 数据模型（TypeScript 接口草案）

- `GameConfig`（初版建议，可调）：
  - `targetFps: 60`
  - `aimIntervalMs: 100`
  - `timeEventIntervalMs: 1000`
- `WaveConfig`：
  - `waveIndex: number`
  - `durationSeconds: number`
  - `spawnIntervalSeconds: number`
  - `spawnCountTotal: number`
  - `enemyId: string`
- `EnemyConfig`：
  - `enemyId: string`
  - `hp: number`
  - `moveSpeed: number`
  - `damage: number`
  - `radius: number`
- `BulletConfig`：
  - `bulletId: string`
  - `speed: number`
  - `damage: number`
  - `lifetimeSeconds: number`
  - `radius: number`
  - `maxRange: number`

### 16.3 算法口径（索敌/碰撞/伤害/回收）

- 索敌：
  - 最近口径：2D 世界坐标欧氏距离
  - 刷新频率：每 `100ms` 重新计算
  - 无目标：不射击
- 碰撞：
  - 判定方式：`Physics2D + Collider2D` 的接触回调（建议 `CircleCollider2D`）
  - 判定频率：由物理系统触发（接触开始/接触中/接触结束）
- 伤害：
  - 敌人对玩家：接触仅代表“可攻击”，实际扣血在 Attack 动画命中帧触发；并带可配置冷却间隔（间隔 `TODO`）
  - 子弹对敌人：命中一次即回收（默认单次命中；穿透不做）
- 回收：
  - 统一重置项：位置/旋转/速度相关临时状态/激活态
