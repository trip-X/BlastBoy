## 2. 技术规范准确性验证

说明：被审查文档未定义传统“API 接口”（HTTP/SDK）。此处将“事件接口/服务接口/模块接口”视为项目内部 API。

### 2.1 接口定义（内部 API）

已具备：

- 事件名与数据形状（`HP_CHANGED/WAVE_CHANGED/TIME_CHANGED/GAME_OVER`）。

缺失导致的问题：

- 缺少事件发布/订阅边界：谁发、何时发、频率、是否允许丢帧合并、是否在暂停时继续。
- 缺少错误处理：事件 payload 非法/空订阅/重复订阅/解绑时机。

### 2.2 数据模型设计

现状：仅有“概念级实体”与“素材命名规范”，没有可落地的数据结构。

建议最低补齐（文档层面，不要求一次到位）：

- `GameConfig`：玩家/敌人/子弹/波次初版参数表与默认值
- `WaveConfig`：波次索引、持续时间或结束条件、刷怪数量、刷怪间隔、刷怪点策略
- `EnemyConfig`：`enemyId/hp/speed/damage/radius`
- `BulletConfig`：`bulletId/speed/damage/lifetime/radius/maxRange`
- 运行时只读状态（供 UI）：`hpCurrent/hpMax/waveIndex/seconds`

### 2.3 业务逻辑描述

清晰点：

- 模块分层方向（`core/services/configs/types → gameplay → ui`）与解耦目标明确。

歧义点：

- “自动锁定最近敌人”的最近定义、刷新频率、目标丢失与切换策略未定义。
- “圆形距离判定”的半径来源与多次命中规则未定义。
- “对象池回收重置”到底重置哪些组件状态未定义。

## 4. 可实施性评估（技术难点/风险）

### 4.1 可实施性结论

- 作为“学习向 MVP”，总体可实施。
- 作为“可交付可验收的实现说明”，目前缺少关键可执行细节，存在落地风险。

### 4.2 关键技术风险（需要补齐口径）

- 性能风险：Web 平台对象数量上限（敌人/子弹）、刷怪策略与对象池上限未定义，容易出现卡顿或泄漏。
- 输入/焦点风险：Web 失焦、键盘按键粘滞、暂停策略未定义，影响可玩性与测试稳定性。
- 解耦落地风险：EventBus 标为“可选”，但核心数据流依赖事件；若落地不统一，后续 UI/玩法耦合会迅速上升。
- 资源与构建风险：未定义构建产物/发布方式/浏览器兼容范围，会导致上线前集中返工。

## 6. 需确认的技术细节问题（封闭式/半封闭式）

### 6.0 已回收的选择结论（需求方反馈）

- Q1：采用 A `EventBus` 作为 UI 与玩法同步机制
- Q2：波次结束条件采用 A 固定时长
- Q3：索敌刷新频率采用 B 固定间隔，间隔为 `100ms`
- Q4：对象池策略为“先动态扩容，后拒绝+静默”，扩容上限为初始 `maxCount` 的 `3x`
- Q5：失焦自动暂停，回到页面恢复
- Q6：浏览器兼容范围为 Chrome + Edge

1. UI 与玩法数据同步机制采用哪一种？
