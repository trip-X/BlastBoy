## 1. 已确定需求（不需要你确认，直接照做）

### 1.1 玩法目标

- 敌人会追击玩家。
- 敌人与玩家发生“接触判定”时，对玩家造成伤害（有伤害间隔）。
- 敌人会被玩家的自动攻击系统选为目标（“最近敌人”）。
- 敌人死亡后要能被回收/清理（未来接对象池；当前至少要有统一销毁入口）。

### 1.2 技术硬约束（来自项目定稿）

- 平台：Web。
- 相机：跟随玩家。
- 碰撞/伤害：使用 `Physics2D + Collider2D`（接触回调/传感器触发）。

### 1.3 现有工程现实（必须适配现状，不做大重构）

- 目录结构以当前仓库为准：脚本在 `assets/Scripts/`，场景在 `assets/Scenes/`。
- 现有实体基类：[EntityManager.ts](file:///d:/development_tool/Cocos/Project/Blaster%20Boy/blasterBoy/assets/Scripts/Base/EntityManager.ts)
  - 已提供 `x/y` 逻辑坐标、`clampPositionInMap()` 与 `applyGridPositionToNode()`，可复用于敌人。
- 现有全局数据单例：[DataManager.ts](file:///d:/development_tool/Cocos/Project/Blaster%20Boy/blasterBoy/assets/Scripts/Runtime/DataManager.ts)
  - 已存地图行列与 `player` 引用（见 BattleManager）。
- 现有事件系统：[EventManager.ts](file:///d:/development_tool/Cocos/Project/Blaster%20Boy/blasterBoy/assets/Scripts/Runtime/EventManager.ts)
  - 如果需要 UI 或调试日志解耦，可用它，但不是强制。

## 2. 敌人“最小可用”需求（建议按这个实现顺序）

### 2.1 EnemyManager（敌人实体组件）

- 建议创建：`assets/Scripts/Enemy/EnemyManager.ts`（如你希望其他目录，我会按“沙盒选项”征求你确认）。
- 敌人应继承 `EntityManager`，并复用 `x/y` 逻辑坐标 → 每帧换算像素坐标。
- 敌人必须能拿到玩家引用（推荐从 `DataManager.Instance.player` 获取）。

### 2.2 追击逻辑

- 每帧计算“从敌人指向玩家”的方向向量，并以固定速度逼近。
- 敌人的移动速度必须可配置（不要写死在代码里）。

### 2.3 接触伤害（脚本圆形判定）

- 判定口径：通过 `Collider2D` 接触回调判定“发生接触/进入攻击范围”。
- 伤害必须有间隔（cooldown），避免每帧扣血。✅ 你已确认：间隔先写成变量，运行时自行调试，不固化数值。
- Collider 的半径/尺寸必须可调（优先 Inspector 暴露或配置化）。

### 2.4 可扩展性最低要求（为后续加波次/多敌人类型做准备）

- 敌人参数不要写死：至少预留 `speed / hp / damage / radius / damageInterval`。
- 敌人类型差异优先通过配置而不是复制脚本。
- 敌人死亡要走统一入口（例如 `onDead()`），后续接对象池时只改一处。

## 4. 沙盒选项（不确定项：请你选择，我不会擅作主张）

> 你只要回复：`4.1 选 B，4.2 选 A ...` 这种格式即可。

### 4.1 圆形判定的“距离单位”用什么？

- A) **Physics2D 世界单位**：Collider2D 的半径/尺寸用 Physics2D 的世界单位配置（推荐，避免混用坐标体系）。
- B) **按像素直觉调参**：保持 Physics2D 默认比例不改，通过 Inspector 调 Collider 尺寸到“手感正确”。
- C) **自定义比例**：调整 Physics2D 的像素比例/参数，让 Collider 尺寸与贴图像素更直观（更容易踩坑）。

### 4.2 接触伤害的“伤害间隔”你希望是多少？

- A) `0.25s`（更紧张，容错更低）
- B) `0.5s`（推荐：节奏与可读性更平衡）
- C) `1.0s`（更宽松，新手更友好）
- D) 你给一个自定义秒数（例如 `0.35s`）

✅ 你已确认：先**写成变量**，你会在运行时自行调试（本阶段不固化具体数值）。

### 4.3 刷怪的“最小实现方式”先选哪个？

- A) **先只生成 1 个敌人**（写死一个测试敌人，验证追击 + 伤害）✅ 你已确认选 A
- B) **地图边缘随机生成**（先做基础刷怪点算法，后面再接波次）
- C) **玩家周围环形生成**（更像生存射击，但更容易出现贴脸）
- D) **地图配置 spawn points**（最可控，但需要改地图数据结构）

### 4.4 敌人类型：你希望第一版实现到什么程度？

- A) **只实现 1 种敌人**（最小可用）✅ 你已确认选 A
- B) **实现 2 种**（例如 `SKELETON_WOODEN`/`SKELETON_IRON`，差异只在数值与贴图）
- C) **只实现 1 种，但把“多类型框架”先搭好**（配置驱动 + 工厂方法）

### 4.5 敌人接触判定半径/尺寸怎么落？

- A) **写死数值**：直接在敌人脚本写死一个半径/尺寸（不推荐，后续调参麻烦）
- B) **写成变量（推荐）**：在敌人脚本/配置中暴露 `colliderRadius` 或 `colliderSize`，你后续自行调试。✅ 你已确认选 B
- C) **按贴图尺寸推导**：根据 `UITransform` 宽高自动计算 Collider 尺寸（更自动，但更容易和美术资源强绑定）

### 4.6 碰撞体形状方案怎么选？

- A) **仅身体碰撞**：只给敌人挂 1 个 `CircleCollider2D`（最简单；攻击=接触伤害）
- B) **仅攻击判定**：只给敌人挂 1 个 `BoxCollider2D`（更像“攻击盒”，但身体接触概念弱）
- C) **混合（身体圆 + 攻击盒）**：敌人同时有“身体圆形”与“攻击盒”，攻击判定更可控。✅ 你已确认选 C

### 4.7 玩家扣血入口怎么落？

- A) **方法调用**：敌人直接调用 `DataManager.Instance.player.takeDamage(damage)`（最直观，但耦合更高）
- B) **事件驱动（推荐）**：敌人 `emit(PLAYER_TAKE_DAMAGE, damage)`，玩家或主控订阅后扣血（更低耦合）✅ 你已确认选 B
- C) **BattleManager 统一仲裁**：敌人只上报“发生接触”，由 BattleManager 处理扣血（主控更集中）

### 4.8 是否把碰撞体设为传感器（Sensor）？

- A) **全部传感器（推荐）**：`sensor=true`，只用回调做判定，不做物理推开。✅ 你已确认选 A
- B) **身体不传感器，攻击盒传感器**：身体用于阻挡/推开，攻击盒只用于触发。
- C) **全部不传感器**：都参与物理响应（更容易抖动/卡边，需要更多调参）。

### 4.9 接触伤害的结算时机选哪个？

- A) **BEGIN_CONTACT + 冷却**：进入接触就结算一次，冷却到期后再次结算。
- B) **STAY_CONTACT + 冷却**：接触保持期间按冷却持续结算（体验更接近“挤压伤害”）。
- C) **攻击命中帧结算**：接触只代表“可攻击”，真正扣血由 Attack 动画命中帧触发（更贴合状态机）。✅ 你已确认选 C
